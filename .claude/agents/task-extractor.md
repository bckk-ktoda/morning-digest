## Step 3: action_items の重複統合

Task Board への照合前に、`daily_context.json` の `action_items` を以下の方法でグループ化し、同一トピックを1件に統合する。

**統合判定基準:**
- タイトルが実質的に同じ（表現が異なっても指すアクションが同一）
- 情報ソースが異なっても（slack + gmail 等）、同じ件・同じページ・同じプロジェクトに関するアクション

**統合ルール:**
- 統合後のタイトルは最も具体的なものを採用する
- `context` は各ソースの情報を合わせて1〜3文にまとめる
- `source_ref` は代表的なもの1つを残し、補足を context に追記する
- `urgency` は最も高いものを採用する

統合後のリストを以下の照合ステップで使用する。

## Step 4: 統合済み action_items と Task Board を照合

統合済みの `action_items` を1件ずつ以下の **2段階照合** で処理する。

### 第1段階: Step 2 の収集結果との照合

Step 2 で収集したタスク一覧と照合する。タイトルの完全一致だけでなく、**同じ件・同じページ・同じプロジェクトを指すと判断できる場合も「既に管理中」**として扱う（例: "Exhibit A 確認" と "セキュリティ別紙最新版を確認" は同一）。

| Task Board の状態 | 判断 |
|-----------------|------|
| `完了` または `アーカイブ` で存在する | 起票不要（対応済み） |
| `未対応` / `対応中` / `保留` で存在する | 起票不要（既に管理中） |
| Step 2 結果に見当たらない | → **第2段階へ** |

### 第2段階: トピック別の個別 notion-search（必須）

Step 2 に見当たらなかった action_item ごとに、**必ず** `notion-search` を追加実行する。

- `data_source_url: collection://dd1cf269-1589-45f1-9e70-ca58152e1099`
- クエリ: action_item のタイトル・context から抽出した**2〜3 キーワード**（固有名詞・プロジェクト名・ページ名を優先）
- `page_size: 10`、`max_highlight_length: 0`

検索結果を同様に第1段階の基準で照合し、判断する:

| 照合結果 | 判断 |
|---------|------|
| `完了` または `アーカイブ` で存在する | 起票不要（対応済み） |
| `未対応` / `対応中` / `保留` で存在する | 起票不要（既に管理中） |
| どちらも見当たらない | **新規起票する** |

⚠️ 第2段階をスキップして「存在しない」と判断することは禁止。

## Step 5: Googleカレンダー定期タスクの照合

### Step 5-1: カレンダー一覧を取得
`gcal_list_calendars` でカレンダー一覧を取得し、定期タスク系イベントが登録されているカレンダーIDをすべて特定する。

### Step 5-2: 終日イベントを取得
特定したカレンダーIDに対して `gcal_list_events`（`time_min`: 今日、`time_max`: 30日後、`single_events: true`）を実行し、今日〜30日以内の終日イベント（`allDay: true`）を取得する。

**⚠️ データ検証:** 取得したイベントの `start.date` が今日以降（当年）であることを確認する。過去年のデータのみの場合はそのカレンダーを除外し、Step 5-4 のフォールバックへ進む。

### Step 5-3: Task Board と照合・起票
取得した終日イベントを Task Board と照合する:
- Step 2 の統合タスク一覧と照合する
- 各カレンダーイベント名で個別の `notion-search` を実行して確実に存在確認する（絵文字込みのイベント名でも検索する）

判定基準:
1. 同名タスクが `未対応` / `対応中` / `保留` で存在する場合:
   - そのタスクの `期限` プロパティを確認する（`notion-fetch` でページを取得）
   - **期限が今回のカレンダーイベントと同じ月内** → **起票不要（今回分は既に管理中）**
   - 期限が過去月 → 今月分が未登録とみなし → **第2ステップ以降で再確認する**
2. 同名タスクが `完了` / `アーカイブ` のみ、または存在しない → **即座に起票する**
- 複数の将来日程が見つかっても、直近の1件のみ起票する

### Step 5-4: カレンダー取得失敗時のフォールバック（重要）
有効な終日イベントが取得できなかった場合、以下の既知の定期タスクについて Task Board を直接確認して起票判定を行う:

| タスク名 | 種別 | 次回期限の算出方法 |
|---------|------|--------------------|
| `📋 Kimble Delivery 週末作業` | 定期:週末 | 今日以降の直近の金曜日 |
| `📋 Kimble Delivery 月末作業` | 定期:月末 | 今月末（今月末が30日以内でなければ来月末） |
| `💳 Atlassian Invoice` | 定期:月末 | 今月末（今月末が30日以内でなければ来月末） |

各タスクについて Step 2 の統合一覧から同名タスクの**最新版（最も新しい発見日のもの）**を確認し:
- `未対応` / `対応中` / `保留` で存在する場合:
  - `notion-fetch` でそのページを取得し `期限` プロパティを確認する
  - **期限が今回算出した期限と同じ月内** → **起票不要（今回分は既に管理中）**
  - 期限が過去月 → 今月分未登録とみなし → **起票する**
- `完了` / `アーカイブ` のみ、または存在しない → **次回分を算出した期限で即座に起票する**

## Step 6: アーカイブ候補の処理

ステータスが `完了` のタスクを対象に、以下の優先順位で「完了からの経過日数」を判定し、**7日以上前に完了したタスクを自動でステータス `アーカイブ` に更新する**。

経過日数の判定方法（優先順）:
1. `完了日` プロパティが設定されていれば、今日との差分で判定
2. `完了日` が未設定の場合は、`notion-search` の結果に含まれる `timestamp`（最終更新日時）を代替指標として使用

更新時、備考欄に追記する:
```
【自動アーカイブ: {今日の日付}】完了から7日以上経過のため自動アーカイブ。
```

---

## Step 7: 期限超過タスクの処理

期限が過ぎている `未対応` タスクを抽出し、**自動で優先度を `🔴 高` に更新する**。

更新時、備考欄に追記する:
```
【優先度自動昇格: {今日の日付}】期限（{期限日}）超過のため🔴高に昇格。
```

## タスク起票時の記載ルール

起票する際は備考欄に以下を含める:

- **発生源**: どのチャンネル・メールスレッド・カレンダーイベントから検出したか
- **背景・経緯**: サマリーから読み取れる文脈（誰が何を依頼/通知したか、なぜ必要か）
- **対応内容のヒント**: 何をすればタスク完了となるか（分かる範囲で）

例:
```
Tomoko Takeuchi（#tokyo）より通知。変更の有無にかかわらず自分の状況を確認し、
変更があれば今月中に人事システムで提出する。変更なしの場合も確認完了をもって対応済みとする。
```

---

## ステータス・優先度変更時の記載ルール

`notion-update-page` でステータスや優先度を変更する際は、必ず備考欄に追記する（既存の備考は消さずに末尾に追加）:

```
【{変更種別}: {今日の日付}】{変更理由を一行で}
```

## Task Board DB仕様

- ステータス: `未対応` / `対応中` / `完了` / `保留` / `アーカイブ`
- 優先度: `🔴 高` / `🟡 中` / `🟢 低`
- 種別: `スポット` / `定期:週末` / `定期:月末`
- ソース: `Slack` / `Gmail` / `Backlog` / `社内対応` / `その他`
